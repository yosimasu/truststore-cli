# Story 2.1: Certificate Chain Completion Service

## Status
Done

## Story
**As a** developer,
**I want** to create an internal service that can take a certificate and build its complete chain by fetching missing issuers,
**so that** the `add` and `rm` commands can operate on a full, validated chain.

## Acceptance Criteria
1. A new internal function/service is created that takes a certificate as input.
2. If the certificate's issuer is not available, it queries a public Certificate Transparency log service (like `crt.sh`).
3. It recursively fetches issuer certificates until a self-signed root is found or no more issuers can be found.
4. The function returns a complete (or most complete possible) certificate chain.
5. The service gracefully handles network errors and cases where the issuer cannot be found in the CT log.

[Source: docs/architecture/04-4-data-models.md]
- **Certificate**: Use Go's standard `crypto/x509.Certificate` struct which includes:
  - `Subject` and `Issuer` for chain relationship building
  - `SerialNumber` for certificate identification
  - `NotBefore`, `NotAfter` (Validity Period)
  - `PublicKey` and `SignatureAlgorithm`
  - `Extensions` (including AIA for potential future use)
- **Relationships**: A `Certificate` can be an issuer for another `Certificate`, forming a chain relationship that this service will complete

[Source: docs/architecture/06-6-external-apis.md]

- **Search Endpoint**: `GET /?CN=<COMMON_NAME>&output=json&exclude=expired` - Returns JSON with certificate `id` field
- **Download Endpoint**: `GET /?d=<ID>` - Returns raw PEM-encoded certificate
- **Authentication**: None required
- **Rate Limits**: No official limits - client must be respectful
- **Integration Notes**: Two-step process (search then download) with resilient error handling for API changes
