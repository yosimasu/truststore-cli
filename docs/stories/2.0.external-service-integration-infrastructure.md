## Dev Notes

### Previous Story Insights
Stories 2.1, 2.2, and 2.3 have established a ChainService interface that depends on external CT log integration. The current implementation likely uses basic HTTP calls without proper resilience patterns. Story 2.0 needs to provide the robust foundation that the existing ChainService can build upon.

### Data Models
[Source: docs/architecture/04-4-data-models.md]
- **Certificate**: Use Go's standard `crypto/x509.Certificate` struct for certificate representation
- **CTLogClient Interface**: Must provide methods for certificate search and download operations
- **Cache Interface**: Should support generic key-value storage for CT log responses with TTL support

### External API Integration
[Source: docs/architecture/06-6-external-apis.md]
- **crt.sh API Integration**: Two-step process for certificate chain completion:
  - **Step 1 (Search)**: `GET /?CN=<COMMON_NAME>&output=json&exclude=expired` 
  - **Step 2 (Download)**: `GET /?d=<ID>` for raw PEM-encoded certificate
- **Base URL**: `https://crt.sh/`
- **Authentication**: None required
- **Rate Limits**: No official limits, but client must be respectful with request patterns
- **Error Handling**: Must be resilient to API changes and handle both search and download step failures

### File Locations

- Create new: `internal/client/interface.go` - CT log client interface definition
- Create new: `internal/client/ctlog.go` - Real CT log client implementation (extend existing)
- Create new: `internal/client/mock.go` - Mock CT log client for testing
- Create new: `internal/client/cache.go` - Caching infrastructure
- Test files: `internal/client/ctlog_test.go`, `internal/client/mock_test.go`, `internal/client/cache_test.go`

### Architecture Patterns
[Source: docs/architecture/02-2-high-level-architecture.md]
- **Interface-based Design**: CTLogClient must be accessed via interface for testability and dependency injection
- **Dependency Injection**: Client services initialized once and injected into consuming services
- **Strategy Pattern**: Different client implementations (real, mock, cached) for different environments

### Technical Constraints
[Source: docs/architecture/11-11-coding-standards.md]
- **Go Version**: Must use Go 1.25.0 managed via `asdf`
- **No CGo**: Pure Go implementation for maximum portability
- **Error Handling**: Never discard errors, use `fmt.Errorf` with `%w` for wrapping
- **Interface Usage**: CTLogClient and caching must be accessed via interfaces
- **Linting**: Must pass `golangci-lint` v1.59.1 before commit

### Testing Requirements
[Source: docs/architecture/12-12-test-strategy-and-standards.md]
- **Framework**: Go's built-in `testing` package  
- **File Convention**: `*_test.go` files co-located with source code
- **Coverage Goal**: Aim for >80% unit test coverage
- **Mock Strategy**: Use httptest for HTTP client testing, interfaces for service mocking
- **Integration Testing**: Use `net/http/httptest` to simulate crt.sh API responses
- **External API Testing**: Mock CT log responses for fast, reliable tests without network dependency

### Security Considerations
- Validate all HTTP responses to prevent injection attacks
- Implement proper TLS verification for external API calls
- Handle sensitive certificate data securely in cache (consider memory-only storage)
- Implement request/response sanitization to prevent log injection
- Use reasonable timeouts to prevent resource exhaustion

### Performance Considerations
- Implement efficient caching to minimize API calls
- Use connection pooling for HTTP client performance
- Consider concurrent request limits to prevent overwhelming CT log APIs
- Implement proper resource cleanup for HTTP connections and cache entries
