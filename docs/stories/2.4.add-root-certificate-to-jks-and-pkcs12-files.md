# Story 2.4: Add Root Certificate to JKS and PKCS12 Files

## Status
Done

## Story
**As a** Java developer,
**I want** to run `truststore add ca.pem --target keystore.jks --target-password=mysecret`,
**so that** I can import a new Certificate Authority into my application's keystore without complex commands.

## Acceptance Criteria
1. The `add` command supports a `--target-password` flag for the destination truststore. When used with `=value` (e.g., `--target-password=mysecret`), the password is provided directly. When used without a value (e.g., `--target-password`), the user is prompted to enter the password interactively.
2. The CLI uses the Certificate Chain Completion Service (Story 2.1) to validate the source certificate.
3. The CLI can add the identified root certificate (from a remote or local source) to a JKS file.
4. The CLI can add the identified root certificate (from a remote or local source) to a PKCS12 file.
5. If the target JKS or PKCS12 file doesn't exist, it is created with the new root certificate.
6. A default alias is automatically generated for the added certificate.
7. A success message is printed, including the alias of the added certificate.
8. Handles incorrect passwords and file write errors gracefully.
9. A loading indicator is displayed during certificate chain completion, validation, and truststore write operations.

## Dev Notes

### Previous Story Insights
Stories 2.2 and 2.3 established comprehensive patterns for certificate addition to PEM files, including chain completion service integration, loading indicators, and user feedback. Story 2.3 specifically implemented local file certificate reading and validation patterns that should be leveraged for consistent behavior.

### Data Models
[Source: docs/architecture/04-4-data-models.md]
- **Certificate**: Use Go's standard `crypto/x509.Certificate` struct for certificate representation
- **Truststore Interface**: JKS and PKCS12 handlers must implement the `Truststore` interface with `AddCertificate(filepath string, cert *x509.Certificate, password string) error` method

### Existing Truststore Handler Implementation
[Source: docs/architecture/08-8-source-tree.md]
- **JKS Handler Location**: `internal/store/jks.go` - Currently implements `ReadCertificates` but has placeholder `AddCertificate` method
- **PKCS12 Handler Location**: `internal/store/pkcs12.go` - Currently implements `ReadCertificates` but has placeholder `AddCertificate` method
- **Interface Definition**: `internal/store/interface.go` defines the `Truststore` interface that both handlers implement

### JKS Library Integration
[Source: docs/architecture/03-3-tech-stack.md]
- **JKS Library**: `github.com/pavlo-v-chernykh/keystore-go` v4.5.0 for reading/writing JKS files
- **Usage Pattern**: Create keystore with `keystore.New()`, load existing with `Load()`, add trusted certificates with `SetTrustedCertificateEntry()`, save with `Store()`
- **Alias Generation**: Use format "cert-{timestamp}" for automatic alias generation

### PKCS12 Library Integration
[Source: docs/architecture/03-3-tech-stack.md]
- **PKCS12 Library**: `software.sslmate.com/src/go-pkcs12` v0.6.0 for reading/writing PKCS12 files
- **Usage Pattern**: Use `pkcs12.Encode()` or `pkcs12.Legacy.Encode()` to create PKCS12 data, then write to file
- **Certificate Addition**: Create new PKCS12 with existing certificates plus new certificate

### Password Handling Security
[Source: docs/architecture/13-13-security.md]
- **Memory Safety**: Passwords must only exist in memory during operations, never logged
- **Interactive Prompts**: Use secure password input that doesn't echo to terminal
- **Command-Line Passwords**: Support but warn about security implications in documentation
