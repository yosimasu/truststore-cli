# Story 2.7: Root Certificate Selection Algorithm Fix

## Status
Ready to Review

## Story
**As a** system administrator,
**I want** the system to correctly identify and select the actual root certificate from certificate chains,
**so that** the proper root CA is added to my truststore instead of intermediate or leaf certificates.

## Acceptance Criteria

**Correct Root Identification:**

1. Enhance existing certificate chain service with `findRootCertificate()` function
2. Root identification logic: Find certificate where subject equals issuer (self-signed root)
3. If no self-signed certificate found, select certificate highest in chain with longest validity period
4. Replace existing naive `chain[len(chain)-1]` selection logic with proper root identification

**Chain Analysis Logic:**
5. For self-signed certificates: Return the certificate itself as both leaf and root
6. For CA-signed certificates: Analyze complete chain to identify actual root certificate
7. Add validation that selected certificate can verify certificates lower in the chain
8. Handle edge cases: incomplete chains, multiple potential roots, cross-signed certificates

**Validation & Testing:**
9. Unit tests cover specific certificate chain scenarios:
   * `example.com`: Should select `CN=DigiCert Global Root G3` (not intermediate)
   * `iot.auomesh.io`: Should select `CN=USERTrust RSA Certification Authority`
   * `mqtt.auomesh.io`: Should select the self-signed `CN=ROOT`

10. Integration tests verify correct root certificates are added to all truststore formats
11. Backward compatibility maintained with existing add/remove command workflows
12. Error handling for malformed or incomplete certificate chains

## Dev Notes

### Implementation Summary

### Actual Implementation Details
**Primary Implementation Location**: `internal/service/chain.go`
- Added `FindRootCertificate(chain []*x509.Certificate) *x509.Certificate` method to existing `ChainService` interface
- Sophisticated root certificate selection algorithm replaces naive `chain[len(chain)-1]` approach
- Maintained full backward compatibility with existing `IsSelfSigned()` and `DetectCertificateType()` methods

**Core Algorithm Implementation**:
1. **Step 1**: Find all self-signed certificates using existing `DetectCertificateType()` method
2. **Step 2**: If self-signed certificates found, use `selectBestRootFromCandidates()` to choose optimal root
3. **Step 3**: If no self-signed certificates, use `selectFallbackRoot()` with longest validity period logic

**Supporting Algorithm Methods**:
- `selectBestRootFromCandidates()`: Selects best root from multiple self-signed candidates using verification count and validity period tie-breaking
- `selectFallbackRoot()`: Fallback selection for incomplete chains using verification capability and validity period
- `countVerifiableCertificates()`: Counts how many certificates in chain can be verified by candidate certificate
- Leverages existing `canVerifyCertificate()` method using Go's `CheckSignatureFrom()` validation

### Code Integration Points
**Updated Files**:
- `internal/app/add.go`: Replaced naive selection in 2 locations (`handleDomainAdd()`, `handleFileAdd()`)

**Integration Pattern**:

### Algorithm Performance Characteristics
**Computational Complexity**: O(n²) where n is chain length (acceptable for typical certificate chains of 2-4 certificates)
**Verification Operations**: Uses Go's standard `CheckSignatureFrom()` for cryptographic validation
**Memory Usage**: Minimal additional memory allocation, operates on existing certificate chain slice
**Execution Time**: Sub-millisecond performance for typical certificate chains

### Edge Case Handling Implementation
**Incomplete Chains**: Fallback logic selects certificate with longest validity period that can verify others
**Multiple Self-Signed Certificates**: Tie-breaking using verification count, then validity period
**Cross-Signed Certificates**: Algorithm handles multiple potential roots by verification capability analysis
**Single Certificate Chains**: Direct return of the single certificate (handles self-signed root scenarios)
