### HTTP Client Specifications
[Source: docs/architecture/03-3-tech-stack.md, docs/architecture/06-6-external-apis.md]
- **Base Technology**: Go's standard `net/http` package
- **Timeout Configuration**: Configurable timeouts (default 15 seconds, matches existing CT Log client)
- **Retry Logic**: Exponential backoff for transient failures, respect rate limits
- **Connection Pooling**: Efficient connection reuse for multiple requests
- **Mock Testing**: `net/http/httptest` package for reliable offline testing

### External API Integration  
[Source: docs/architecture/06-6-external-apis.md]
- **CT Log Integration**: Must maintain compatibility with existing crt.sh API usage
- **Rate Limiting**: Good citizen behavior, avoid aggressive polling
- **Error Resilience**: Handle API changes gracefully, comprehensive error classification
- **Network Detection**: Detect offline scenarios and provide graceful fallbacks

### Technical Constraints
[Source: docs/architecture/11-11-coding-standards.md]
- **Go Version**: Must use Go 1.25.0 managed via `asdf`
- **No CGo**: Pure Go implementation for maximum portability
- **Error Handling**: Never discard errors, use `fmt.Errorf` with `%w` for wrapping HTTP errors
- **Interface Usage**: HTTP clients must be used via interfaces for dependency injection
- **Standardization**: Establish patterns that can be used across future API integrations

### Testing Requirements
[Source: docs/architecture/12-12-test-strategy-and-standards.md]
- **Framework**: Go's built-in `testing` package
- **File Convention**: `*_test.go` files co-located with source code
- **Coverage Goal**: Aim for >80% unit test coverage
- **Mock Strategy**: Use `net/http/httptest` for HTTP client mocking, no external mocking libraries
- **Integration Testing**: Support both online (live API) and offline (mock server) test modes
- **Error Testing**: Comprehensive testing of timeout, network failure, and server error scenarios

### Security Considerations
- Implement secure default timeout values to prevent hanging connections
- Validate HTTP response content to prevent injection attacks
- Handle TLS certificate validation properly for HTTPS connections
- Ensure proper cleanup of HTTP connections to prevent resource leaks
- Log HTTP errors appropriately without exposing sensitive information
