# Story 1.4: List Certificates from JKS and PKCS12 files

## Status
Done

## Story
**As a** Java developer,
**I want** to run `truststore list keystore.jks --password mysecret`,
**so that** I can inspect the contents of my application's keystore without using `keytool`.

## Acceptance Criteria
1. The `list` command accepts a `--password` flag for protected truststores.
2. The CLI can successfully read and parse certificates from a JKS file.
3. The CLI can successfully read and parse certificates from a PKCS12 file.
4. The output format is consistent with all other `list` commands.
5. Clear error messages are provided for incorrect passwords or corrupted files.

### Data Models
[Source: docs/architecture/04-4-data-models.md]
- **Certificate**: Use Go's standard `crypto/x509.Certificate` struct which includes:
  - `Subject` and `Issuer`  
  - `SerialNumber`
  - `NotBefore`, `NotAfter` (Validity Period)
  - `PublicKey` and `SignatureAlgorithm`
  - `Extensions` (including AIA for potential future use)
- **Truststore Interface**: Must implement common interface with methods like `ReadCertificates()` to provide generic certificate container abstraction regardless of file format
- **Relationships**: A `Certificate` can be an issuer for another `Certificate`, forming a chain relationship

### Architecture Patterns
[Source: docs/architecture/02-2-high-level-architecture.md]
- **Strategy Pattern**: Each truststore format (PEM, JKS, PKCS12) will be handled by a specific "strategy" (a struct that implements a common `Truststore` interface with methods like `Read`, `Add`, `Remove`)
- **Dependency Injection**: Core services will be initialized once and passed into commands that need them for improved testability and clear dependency management
- **Command Pattern**: Cobra library inherently uses command pattern to encapsulate all information needed to perform actions

### Error Handling Requirements
[Source: docs/architecture/10-10-error-handling-strategy.md]
- **Error Model**: Use Go's standard `error` interface with `fmt.Errorf` and `%w` for wrapping
- **Password Errors**: Incorrect password errors will be caught and presented with clear, actionable messages without exposing passwords
- **File I/O Errors**: Errors like "file not found", "permission denied", or "corrupted file" will be presented with clear messages
- **Output**: User-facing error messages printed to `stderr`
- **Error Propagation**: Errors passed up call stack until they reach top-level command handler
- **Security**: No sensitive information (like passwords) will ever be printed to stdout or stderr

### Coding Standards
[Source: docs/architecture/11-11-coding-standards.md]
- **Error Handling**: Never discard errors with `_`, use `fmt.Errorf` with `%w` for wrapping
- **Naming**: Follow Go community conventions (camelCase for locals, PascalCase for exports)
- **No CGo**: Must remain pure Go for maximum portability
- **Interfaces**: Use interfaces for services to support dependency injection and mocking
- **Use Interfaces for Services**: All major services and data handlers (e.g., `JksHandler`, `Pkcs12Handler`) must be used via interfaces

### Testing Requirements
[Source: docs/architecture/12-12-test-strategy-and-standards.md]
- **Framework**: Go's built-in `testing` package
- **File Convention**: `*_test.go` files co-located with source code
- **Coverage Goal**: Aim for >80% unit test coverage
- **Test Data**: Store sample JKS and PKCS12 files in `testdata` subdirectory within the package being tested
- **JKS Test Data Generation**: Use keytool (Java temurin-17.0.16+8 via asdf) to generate test JKS files with certificates:
  - `keytool -genkeypair -alias testcert -keyalg RSA -keysize 2048 -validity 365 -keystore test.jks -storepass testpass -keypass testpass -dname "CN=test-cert"`
  - `keytool -delete -alias testcert -keystore test.jks -storepass testpass` (remove private key, keep only cert)
  - For password-free variant: `keytool -storepasswd -keystore test.jks -storepass testpass -new ""`
- **PKCS12 Test Data Generation**: Use OpenSSL to generate test certificates and PKCS12 files:
  - `openssl req -x509 -newkey rsa:2048 -keyout temp-key.pem -out certificate.crt -days 365 -nodes -subj "/CN=test-cert"`
  - `openssl pkcs12 -export -out test.p12 -nokeys -in certificate.crt -password pass:testpass`
  - `openssl pkcs12 -export -out test-nopass.p12 -nokeys -in certificate.crt -passout pass:`
  - Clean up temporary key file: `rm temp-key.pem`
  - Generate both password-protected and unprotected variants for comprehensive testing
- **Mocking**: Use Go interfaces for dependency mocking (no external libraries)
- **Test Organization**: Unit tests in same package as code being tested
